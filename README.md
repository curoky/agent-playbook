# 个人编码规范

本仓库用于**维护一份「我个人通用的编码规范」**。仓库没有应用源码，**唯一产物就是 [`guidelines/CLAUDE.md`](./guidelines/CLAUDE.md) 本身**——它是一份给 AI 助手（及人）阅读的编码与协作规范，可被拷贝或链接到其他项目中使用。

> 给后续 agent 的提示：本仓库的「任务」不是写应用代码，而是持续打磨 `guidelines/CLAUDE.md` 这份规范文档。对本仓库的改动，几乎都是在编辑它。根目录的 `CLAUDE.md` 只是本仓库的简短指引，不要与产物混淆。

## 仓库结构

- `guidelines/CLAUDE.md` —— **产物主文件**：高频遵守的规范（拷到其他项目即可作为该项目的 `CLAUDE.md`）。
- `guidelines/.agent/` —— **按需查阅的明细**，由主文件用相对路径链接引用：
  - `libraries.md` —— JS/Python 分场景库选型表。
  - `toolchain.md` —— 统一工具链（包管理、Lint、类型检查、测试、构建、pre-commit）。
  - `coding-practices.md` —— 编码实践完整要点（命名、函数与模块、类型与错误、异步、性能、注释、测试、安全）。
  - `project.md` —— 项目与工程化（项目结构、配置与环境管理、日志与可观测性）。
  - `versioning.md` —— 版本与协作（提交规范、SemVer、changelog、依赖治理、CI/CD）。
- `CLAUDE.md`（根目录）—— 本仓库的简短指引，指向产物与本说明。
- `README.md` —— 本文件，仓库说明。

> 拷贝到其他项目时：把整个 `guidelines/` 目录一并带上（含 `.agent/`），否则明细链接会指向不存在的文件。主文件本身可独立阅读，明细按需点链接查看。

## 产物结构导览

`guidelines/CLAUDE.md` 按五个领域组织，每个领域下是若干条目（`### N. xxx`），条目统一为「**核心原则**（一句导语）+ 要点列表」的风格：

| 领域 | 定位（管什么） | 主要内容 |
| --- | --- | --- |
| 一、技术栈与工具基线 | 用什么 | 优先复用开源组件（含 JS/Python 选型表）、现代语言版本与语法、统一工具链 |
| 二、编码实践 | 怎么写 | 命名风格、函数与模块设计、类型安全与错误处理、异步并发、性能、注释文档、测试、安全编码 |
| 三、项目与工程化 | 项目怎么搭 | 项目结构与组织、配置与环境管理、日志与可观测性 |
| 四、版本与协作 | 怎么协作发布 | 提交规范、语义化版本、变更日志、依赖治理、CI/CD 流水线 |
| 五、与 AI 协作 | AI 如何工作 | 思考在先、简单优先、外科式改动、目标驱动执行、沟通与交付（行为准则，优先级最高） |

> 具体技术栈以 **JavaScript/TypeScript** 与 **Python** 为准；其他语言沿用各领域中与语言无关的通用原则，并套用对应生态的等价工具。

## 迭代与维护原则

修改 `guidelines/CLAUDE.md` 时遵循：

- **保持风格一致**：新条目沿用「核心原则 + 要点列表」格式；新增条目按所属领域归类并顺延编号（`### N.`）。
- **具体而非泛泛**：给出具体的库名、版本号、命令、配置项，而非空泛建议；库一律用 GitHub 链接（标准库用官方文档链接）。
- **增删条目**：跨领域调整时同步更新顶部简介中的领域描述，以及条目间的交叉引用文字（如「见『类型安全与错误处理』」）和指向 `.agent/` 的明细链接，避免引用失效。
- **核实时效性**：涉及推荐库、语言/工具版本时，定期核实其最新发布时间与维护状态，及时替换停更依赖、校准版本号（文档中标注了「截至 2026-06」的具体版本，落地时以官方最新稳定版为准）。
- **控制篇幅**：单个条目聚焦单一主题；要点避免冗余铺垫与重复举例，保留可执行信息。

## 使用方式

将这份规范应用到其他项目：

- **拷贝**：把整个 `guidelines/` 目录复制到目标项目（含 `.agent/`），用其中的 `CLAUDE.md` 作为该项目的 `CLAUDE.md`；明细用相对路径链接，故必须连目录带上。
- **同步**：也可用 git submodule / 软链等方式引入 `guidelines/`，便于统一更新。

应用后按目标项目实际情况裁剪（如只用其中某些领域、调整版本基线）。

# 代码重构（按重写的思路）

> 本 skill 是对 [`ai-collaboration.md`](../../rules/ai-collaboration.md) 「外科式改动」「简单优先」「目标驱动执行」与 [`main.md`](../../rules/main.md) 编码实践的具体化，专用于**重构**场景。执行前先加载这些规则，冲突时以本 skill 的重构判据为准。

## 核心理念

**重构的验收标准是：如果按最终预期从零重写，重构后的代码要和重写的一样干净——没有任何历史包袱。**

不要把重构做成「在旧代码上打补丁」。改完之后，读代码的人应当看不出它是「从 A 改到 B 再改到 C」演化来的，只应看到「本该如此」的 C。

## 必须清除的历史包袱

- **中间态与迁移痕迹**：从 A → B → C 的演化过程不留在代码里——不保留「为什么从 A 改到 B、又为什么改到 C」的注释、被注释掉的旧实现、`// 旧逻辑`/`// deprecated`/`// 兼容老版本` 之类的说明。历史用 git 记录，不用代码承载。
- **兼容层与回退分支**：不保留「兼容 A」「兼容 B」的 if 分支、feature flag、双写、版本判断、adapter/shim。既然目标是 C，就只留 C，删掉所有为过渡而存在的代码路径。
- **孤儿代码**：清理因重构不再被引用的 import、变量、函数、类型、常量、配置项、依赖、测试与死分支。重构后不应残留任何「只有旧路径才用得到」的东西。
- **过时的名字**：函数/变量/文件名若还带着旧语义（`handleV2`、`newXxx`、`tmpFix`），一并改成描述最终职责的名字。

## 简洁优先（个人项目默认取向）

大多数项目是自用的，优先「代码简洁明了」而非「通用可配置」：

- **能 hardcode 就 hardcode**：把只有一处用、且短期不会变的配置项、开关、参数直接写死进代码，删掉为它们准备的配置读取、schema、环境变量与默认值回退。不为「将来可能要改」保留配置化。
- **删掉投机性灵活度**：无实际第二个调用方的抽象、泛型、策略模式、插件点、只被一处使用的参数——全部内联收敛。
- **自检**：「如果现在从零写，会写成这样吗？」不会，就继续简化，直到答案为「会」。

## 系统性降复杂度（每次重构都要评估）

动手前系统评估以下三条能否显著降低代码量与复杂度，能则优先采用（引入依赖遵循 [`main.md`](../../rules/main.md) 「优先复用成熟的开源组件」的选型判据）：

- **引入开源库**：手写的能力（参数解析、日期、校验、重试、并发限流、状态机等）是否有成熟、维护良好的库能大幅替代？能大幅简化代码或 API 明显更不易出错的（如 `zod`、`pydantic`、`typer`）优先替换手写实现。
- **引入/统一工具链**：格式化、lint、类型检查、测试、构建能否统一到标准工具链，替换掉自研脚本或散落配置。
- **重新抽象**：现有结构是否本就该换一套模型？该合并的重复逻辑、该拆分的巨型函数/文件、该扁平化的深层嵌套、该用数据驱动替代的大量分支——从整体重新设计而非局部缝补。

## 执行流程

1. **定目标形态**：先明确「最终 C 应该长什么样」——数据模型、模块边界、对外 API、依赖清单。这是重写蓝图，也是验收基准。
2. **评估降复杂度**：按上节三条评估是否引入库/工具链/重新抽象，把结论作为蓝图的一部分（涉及新依赖或较大结构调整时，先与用户确认再动手）。
3. **建立安全网**：重构前确保有覆盖关键行为的测试；缺失则先补能验证「重构前后行为一致」的测试（针对公共行为，不绑定内部实现）。
4. **按蓝图重写**：以 C 的形态改写，同步删除所有中间态、兼容层、孤儿代码与投机配置，不做与本次重构无关的额外改动。
5. **验证**：跑「格式 → lint → 类型 → 测试」全绿；对照蓝图确认行为不变、无遗留包袱。
6. **交付说明**：说清最终形态是什么、删掉了哪些历史包袱/兼容逻辑、引入或移除了哪些依赖、如何验证（跑了哪些命令）。

## 保持行为等价的边界

「按重写的思路」指的是**代码形态**要像重写，不是**擅自改变功能**。除非用户明确要求，重构须保持对外可观察行为不变；发现顺带能修的 bug 或想变更行为时，先指出并与用户确认，不静默夹带。

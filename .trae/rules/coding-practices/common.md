# 编码实践（通用）

> 本文件是主规范「编码实践」领域的明细，列出各主题的**核心原则与语言无关要点**；主文件（[`main.md`](../main.md)）只保留每节的核心原则与关键禁令。
>
> 各语言的**特有写法、推荐语法与禁止项**见同目录分语言文件，编辑对应源文件时自动生效：[`js.md`](./js.md)、[`python.md`](./python.md)、[`go.md`](./go.md)、[`cpp.md`](./cpp.md)。

## 0. 语言版本与语法

**核心原则**：用主流支持的较新版本（版本基线总表见 [`main.md`](../main.md)「技术栈与工具基线 · 2」）；优先用新语法简化代码、提升可读性与类型安全，但不为用而用——新写法反而更难懂时（如深度嵌套 `:=`、把简单分支硬写成 `match`、过度类型体操）选直观写法；不用已 EOL 版本，不用实验性/非 Stage 4 语法于生产。版本号须显式锁定以保团队一致。

> 各语言的版本锁定方式、推荐语法与禁止写法见分语言文件的「§0 语言版本与语法」。

## 1. 命名与代码风格

**核心原则**：命名即文档；风格统一交给工具（`biome` / `ruff` / `gofmt` / `clang-format`）强制，人只关注命名与表达意图。

- **命名表意**：用完整、可检索的名字，避免缩写与单字母（循环计数器等惯例除外）；布尔值用 `is`/`has`/`should` 前缀；函数名用动词短语，变量/类用名词短语。
- **避免魔法值**：字面量数字/字符串提取为有名常量或枚举（各语言惯用法见分语言文件）。
- **格式不靠手写**：缩进、引号、分号、import 排序等全部交给格式化工具自动处理，不在 review 中讨论格式问题。

> 各语言的大小写惯例与魔法值提取方式见分语言文件「§1 命名与代码风格」。

## 2. 函数与模块设计

**核心原则**：小而专一、低耦合高内聚；依赖于抽象而非实现，便于测试与复用。

- **单一职责**：一个函数只做一件事；函数过长或有多个「阶段性注释」时，拆分为更小的具名函数。
- **参数精简**：参数超过 3 个时改用对象/具名参数；避免布尔陷阱参数（用枚举或拆分函数）。
- **优先纯函数**：核心逻辑写成无副作用的纯函数，把 I/O、随机、时间等副作用推到边界。
- **明确公共 API**：模块通过显式导出暴露最小必要接口。
- **依赖倒置**：高层逻辑依赖接口/协议，由外部注入实现，避免在深处直接 `new` 具体依赖。
- **避免循环依赖**：模块依赖保持单向、分层清晰；出现循环引用时通过提取公共模块或反转依赖解决。

> 各语言的具名参数、公共 API 暴露、接口/依赖注入的具体写法见分语言文件「§2 函数与模块设计」。

## 3. 类型安全与错误处理

**核心原则**：用类型表达约束、让非法状态不可表示；错误必须显式、就近、可追溯地处理，禁止静默吞掉。

**类型安全**：

- **不使用逃逸类型**：避免绕过类型系统的「万能类型」，用具体类型、泛型或受约束的多态替代。
- **在边界校验外部输入**：API 响应、用户输入、配置、环境变量等不可信数据，必须在进入系统时用 Schema 校验后转为内部可信类型。
- **让非法状态不可表示**：优先用联合类型/可辨识联合等手段建模互斥状态，避免「多个布尔标志组合出非法态」。
- **避免「可能不存在」的隐式约定**：对可空/可缺失的值用显式类型标注，不靠隐式约定。

**错误处理**：

- **可预期错误用值表达**：业务上可预期的失败（找不到、校验失败、外部调用失败）优先用返回值表达，而非异常或易忽略的错误码。
- **异常用于不可恢复错误**：异常机制仅用于真正异常、不可预期的情况；捕获时必须捕获**具体类型**，禁止裸捕获后不处理。
- **不静默吞错**：捕获后必须做有意义的处理（恢复、转换、上报、带上下文重抛），禁止空 catch、禁止只打日志后继续。
- **保留上下文**：重新抛出/包装错误时携带原始错误（cause/根因），不丢失定位信息。
- **资源清理可靠**：用语言提供的确定性释放机制确保文件/连接/锁等资源释放，不依赖手动调用。

> 各语言的逃逸类型禁令、校验库、错误表达方式（`Result`/`error`/`expected`）、上下文保留与资源管理写法见分语言文件「§3 类型安全与错误处理」。

## 4. 异步与并发

**核心原则**：异步代码必须显式管理生命周期与错误；不阻塞、不泄漏、不丢异常。

- **统一异步模型**：每个并发单元都有明确的退出条件，不「开后无人管」。
- **并发要并行**：相互独立的异步任务并发执行，不要串行等待。
- **限制并发度**：批量并发时限制上限，避免压垮下游或耗尽连接/句柄。
- **不吞异步异常**：每个异步任务的失败都必须被处理；禁止「发射后不管」的悬空任务/泄漏并发单元。
- **支持取消与超时**：长任务支持取消与超时，避免无限等待。
- **保护共享状态**：并发访问共享可变状态时用锁/队列/不可变数据，避免竞态；优先用消息传递而非共享内存。

> 各语言的异步原语（`async`/`await`、goroutine、`std::jthread`）、并行与限流工具、取消机制见分语言文件「§4 异步与并发」。

## 5. 性能与优化

**核心原则**：先用 profiler / benchmark 定位真实瓶颈再优化，不靠直觉做过早优化。

- **基准可复现**：关键算法/接口写 benchmark，优化前后对比并纳入回归；用 profiler 定位 CPU/内存热点。
- **避免常见浪费**：N+1 查询、循环内重复计算/IO、不必要的深拷贝、未加索引的查询；批量与缓存优先于逐条处理。
- **按瓶颈选并发模型**：IO 密集用异步并发（见「§4 异步与并发」），CPU 密集用 worker / 多进程 / 线程池，不阻塞事件循环 / 主线程。
- **缓存有度**：缓存须明确失效策略与一致性边界，避免难调试的陈旧数据。

> 各语言的 benchmark/profiler 工具与具体优化手段（预分配、移动语义、对象复用等）见分语言文件「§5 性能与优化」。

## 6. 注释与文档

**核心原则**：注释解释「为什么」，而非「做了什么」；代码本身说明「做了什么」。

- **写 why 不写 what**：解释意图、权衡、坑与非显而易见的约束；不要复述代码字面逻辑。
- **公共 API 必须有文档**：导出的函数/类/模块写文档注释（各语言文档注释风格见分语言文件）。
- **标注临时与风险**：用统一标记 `TODO`/`FIXME`/`HACK` 并附负责人或 issue 链接；对已知坑、绕过方案写明原因。
- **注释与代码同步**：修改代码必须同步更新相关注释/文档，过时注释比没有注释更有害；删除被注释掉的「死代码」（依赖版本控制找回）。
- **不写废话注释**：禁止 `i++ // 自增` 这类复述型注释。

## 7. 测试规范

**核心原则**：测试是行为契约；测公共行为而非内部实现，保证可重构、可信任。

- **结构清晰**：遵循 Arrange-Act-Assert（准备-执行-断言）；用例名描述「在什么条件下、期望什么行为」。
- **测行为非实现**：针对公共接口与可观察行为断言，避免断言私有细节，减少重构时的脆性。
- **隔离与确定性**：单测不依赖网络/真实 DB/时钟/全局状态；外部依赖用 mock/stub/fake，时间用可注入的时钟，保证可重复、可并行。
- **覆盖关键路径**：优先覆盖核心逻辑、分支与边界条件（空、极值、错误路径），不盲目追求 100% 覆盖率；覆盖率作为参考而非目标。
- **回归即用例**：修复 bug 时先写一个能复现该 bug 的失败用例，再修复使其通过。

> 各语言的测试框架（`vitest`/`pytest`/`testing`/`Catch2`）、mock 工具与表驱动/参数化写法见分语言文件「§7 测试规范」。

## 8. 安全编码

**核心原则**：默认不信任一切外部输入；最小权限、纵深防御。

- **校验并转义输入**：所有外部输入先校验再使用；按输出上下文转义（HTML/SQL/Shell/URL），防注入。
- **杜绝注入**：SQL 用参数化查询或 ORM，禁止字符串拼接 SQL；执行子进程传参数数组而非拼接命令行字符串。
- **最小权限**：进程、令牌、数据库账号、CI 凭据均按最小必要权限授予；敏感操作做鉴权与审计。
- **安全使用加密**：密码用专用算法哈希（如 `argon2`/`bcrypt`，不用 MD5/SHA1）；随机数用密码学安全源，不用通用随机数 API。
- **依赖与供应链安全**：见「版本与协作 · 依赖治理」，CI 强制漏洞扫描与升级。

> 各语言的校验库、参数化/转义 API、CSPRNG 与内存安全（C++）等具体手段见分语言文件「§8 安全编码」。
